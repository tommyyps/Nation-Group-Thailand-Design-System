# สรุปการอัปเดต / แก้ไขโปรเจกต์ Nation Group Thailand Design System

เอกสารนี้จัดตาม **วันที่ → หัวข้อใหญ่ → หัวข้อย่อยและรายละเอียด** เพื่อใช้เปรียบเทียบกับสำเนาโค้ดอื่น (เช่น Repo บน GitHub) โดยดูว่า **เนื้องานตรงกันหรือไม่**

---

## แหล่งอ้างอิงวันที่

- **วันที่ทุกข้อในไฟล์นี้อ้างอิงจากการบันทึกในเอกสารฉบับนี้และบริบทงานในโปรเจกต์เท่านั้น** (รอบแก้ไข / ลำดับที่สรุปไว้ที่นี่)
- **ไม่** ใช้วันที่จาก GitHub, `git log`, หรือประวัติ commit เป็นต้นฉบับของวันที่ในตารางสรุป
- หากต้องการแยกหัวข้อตามวันจริงของทีม ให้แก้เฉพาะหัวข้อ `## วันที่ …` ในไฟล์นี้ให้ตรงกับที่ทีมตกลง — ยังคงอ้างอิงจาก **ในเอกสารนี้** ไม่ต้องดึงจาก Git

---

## บันทึกคู่กับการพัฒนา (ทำไปเขียนไป)

**ทำได้** — ไฟล์นี้เป็น Markdown ที่แก้มือได้ตลอด ไม่มีสคริปต์ดึงจาก Git ให้อัตโนมัติ ดังนั้นการให้สรุป “ตามงานจริง” ต้องเกิดจาก **ทีมหรือผู้ช่วย (Cursor) ใส่ข้อความลงมา** ตามรอบที่ตกลง

### เวลาบันทึกที่ตกลง (รายวัน)

ทีมใช้ **สองรอบต่อวัน** เพื่อสรุปงานลงไฟล์นี้ (หรือสั่ง Cursor ให้ช่วยสรุปตามรอบ):

| รอบ | เวลา | แนวทาง |
|------|------|--------|
| กลางวัน | **12:00 น.** | สรุปงานช่วงเช้า — เติมใต้หัวข้อวันที่วันนี้ หรือเปิดเซสชัน Cursor แล้วสั่งให้อัปเดตเอกสาร |
| ปิดวัน | **17:30 น.** | สรุปทั้งวัน — ปิดท้ายรายการของวันที่นั้นให้ครบก่อนเลิกงาน |

- **เขตเวลา:** ใช้เวลาท้องถิ่นของทีม (เช่นไทย) — ถ้าทำงานข้ามโซนให้ระบุในเอกสารทีมแยก
- **ไม่มีการเตือนอัตโนมัติจากไฟล์นี้** — แนะนำตั้งปฏิทิน / การแจ้งเตือนระบบ หรือนิสัยคู่กับพักกลางวันและเลิกงาน

### แนวทางที่ใช้ง่ายกับ Cursor

- **จบงาน / จบเซสชัน:** สั่งชัด ๆ เช่น *「อัปเดต `docs/สรุปการอัปเดตโปรเจกต์.md` ด้วยวันนี้ สรุปเป็นหัวข้อย่อยว่าแก้อะไรบ้าง」* — ผู้ช่วยจะเพิ่มหัวข้อ `## N. วันที่ …` (หรือเติมใต้วันที่เดิมถ้าเป็นวันเดียวกัน) ให้สอดคล้องรูปแบบด้านล่าง
- **ให้เป็นกฎโปรเจกต์:** ใส่ใน Project Rules / `.cursor/rules` หรือ `AGENTS.md` ว่า *หลังงานที่แตะหลายไฟล์หรือเปลี่ยนพฤติกรรมสำคัญ ให้สรุปลงเอกสารนี้* — ผู้ช่วยจะทำตามเมื่อกฎระบุไว้
- **รูปแบบเดิม:** หัวข้อใหญ่ตามวัน → หัวข้อย่อยตามหน้า/โมดูล (`ColorPlayground`, `generate-tokens.mjs` ฯลฯ) → bullet สั้น ๆ ว่าเปลี่ยนอะไร ทำไม (ถ้าจำเป็น)

### ข้อจำกัดที่ควรรู้

- ผู้ช่วย **ไม่** อัปเดตไฟล์นี้เองทุกครั้งที่แก้โค้ด ถ้า **ไม่ได้รับคำสั่งหรือกฎโปรเจกต์** — เพื่อไม่ให้เอกสารเด้งโดยไม่ตั้งใจหรือขัดกับวันที่ที่ทีมใช้จริง

## 1. วันที่ 1 เมษายน 2568 (2026-04-01)

### หน้า 03 Typography (`TypographyPlayground`)

#### โครงสร้างและลำดับเนื้อหา
- ย้ายบล็อก **「ตัวอย่างการใช้งานจริงบนหน้าเว็บ」** ขึ้นมาเป็นลำดับแรกก่อนสเปซิเมนต์
- แยกสเปซิเมนต์เป็นการ์ด 3 ประเภท: **Heading**, **Text**, **Article** — แต่ละใบมีตัวอย่างโค้ดอยู่ในการ์ดเดียวกัน (ไม่รวมโค้ดกองล่างแบบเดิม)
- แถบหั้วการ์ด: ชื่อซ้าย + **dropdown เลือกขนาด** ขวา (Heading: H1–H6; Text/Article: XL–XS) สั่งทั้งพรีวิวและ snippet โค้ด

#### UI ตามดีไซน์ (การ์ดเดียว)
- การ์ด: แถบหัว → พื้นพรีวิวขาว → บล็อกโค้ดมืดติดด้านล่าง มุมโค้งของกรอบนอกเดียว
- **Label + meta** สเปซิเมนต์: อยู่**บรรทัดเดียวกัน** และอยู่**เหนือ**ตัวอย่างฟอนต์ (ไม่คอลัมน์ซ้าย)

#### แก้บั๊ก
- ตัวอย่างจริง: คอลัมน์บทความใช้ `articleWeight` กับ `group="article"` ให้สอดคล้องกับน้ำหนัก Article

---

### Border radius (การ์ด / เซกชันภายนอก)

#### การปรับตามรอบ
- เคยตั้งการ์ดหลักเป็น `var(--radius-none)` แล้วคืนเป็นโค้งตามดีไซน์
- **ปัจจุบัน (ตามสรุปในเอกสารนี้):** `.playground-preview`, `.playground-controls`, `.code-block` (แยกกล่อง), `.home-card` ใช้ **`var(--radius-md)`**
- คอมโพเนนต์ย่อยในแถบควบคุม: `control-select` / input ยังใช้ `--radius-md`; ปุ่มคัดลอกใน code block ใช้ `--radius-sm`

#### โค้ดซ้อนในการ์ด Typography / Playground
- `.playground-preview-code-wrap .code-block` ใช้ `border-radius: 0` และไม่มีขอบซ้ำ — ให้มุมมาจาก `.playground-preview` ภายนอก

---

### Playground ทุกหน้า — ความสม่ำเสมอ

#### พรีวิวและโค้ดในการ์ดเดียว
- **`PlaygroundShell`:** ห่อ `playground-preview-pane` รอบ canvas; ย้าย **บล็อกโค้ดเข้าใน** `.playground-preview` ภายใต้ `.playground-preview-code-wrap` (เหมือน Typography)
- สไตล์ร่วมใน **`PlaygroundShell.css`**

#### หัวข้อพรีวิว
- เพิ่ม prop **`previewLabel`** (ค่าเริ่มต้น `ตัวอย่าง`) — แต่ละหน้า playground ตั้งชื่อเช่น `ตัวอย่าง — 11 Buttons` ฯลฯ

---

### สไตล์แถบโค้ดและสเปซิเมนต์

#### Code block header
- `padding` เท่ากับแถบหั้วการ์ด Typography (`typography-specimen-card-header`)
- `font-size` เท่ากับ `typography-specimen-card-title` (`--font-size-16`)

#### Typography specimen (label / meta)
- ขนาดตัวอักษร meta ให้เท่ากับ label ในแผงพรีวิว (`--font-size-14`)
- **Font weight:** Regular (`--font-weight-regular`) ทั้ง label และ meta

---

### แถบควบคุม (คุณสมบัติ) — สีแบรนด์

#### Checkbox / radio / range
- ใน `.playground-controls`: `accent-color: var(--color-brand)`
- Range: สไตล์ track/thumb สำหรับ WebKit และ Firefox ใช้สีแบรนด์และเส้นขอบ token — ไม่พึ่งสี accent ของระบบปฏิบัติการ

---

### การ์ด Content (Card) — พื้นหลัง “ล้น” ที่มุม

#### สาเหตุและแก้ไข
- **Card:** `border-radius` ใช้ token `var(--radius-md)` สอดคล้อง playground; เพิ่ม **`overflow: hidden`**
- **`playground-preview-pane` / `playground-canvas`:** `overflow: hidden` และ canvas มี `max-width: 100%`
- **CardPlayground (โหมด elevated):** wrapper ที่มี `boxShadow` ใส่ `border-radius` + `overflow: hidden` ให้ตรงกับการ์ด

---

### เอกสาร — `doc-lead`

#### Line height
- ใช้ตัวแปรคู่กับขนาดอยู่แล้วใน `DocContent.css` (`--font-size-doc-lead` + `--line-height-doc-lead`)
- แก้ค่า token: จาก **21px** (แคบเกินกับ 18px หลายบรรทัดภาษาไทย) เป็น **`var(--line-height-27)`** (~1.5×)
- อัปเดต **`scripts/generate-tokens.mjs`** ให้ emit ค่าเดียวกันเมื่อรัน generate

---

### เอกสารอธิบาย (ไม่ใช่การเปลี่ยนโค้ด)

- คำอธิบายเรื่อง selector CSS แบบ `.parent .child` สำหรับ `.typography-specimen-preview-pane` (ขอบเขตสไตล์และ specificity)
